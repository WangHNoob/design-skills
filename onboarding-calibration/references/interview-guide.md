# 入职校准 · 访谈问题库

和策划共建产出模板时用。原则：**先看真东西（知识库样例）→ 再问要什么 → 呈现草稿 → 改到认可**。一次聚焦一个 skill，别铺开。

## 通用开场（对任意 skill）

1. 展示当前 `output-template.md`，直说："这是占位模板，包作者随手造的，我们来把它换成你们真正在用的。"
2. 拉 1–2 个知识库真实产物当引子（只读、只参考），问：
   - "你们真实的 <该类文档/配表> 一般长什么样？和这个样例比，差在哪？"
   - "有没有你们现在手写文档时**固定会有**的章节，这里缺了？"
   - "这个模板里哪些章节/字段，你们其实**从来不填**？"（→ 删）

## 通用收尾问题（定稿前必问）

- "每个保留章节，一句话说清它要写什么（给占位提示语）？"
- "有没有**必须先有、后面才好写**的顺序要求？"（决定章节顺序）
- "产出给谁看？"（决定详略与术语——但注意：详略偏好若与模板无关，归 PROJECT_PROFILE，不塞模板）

## 按 skill 类型的重点问题

### 文档类：system-design / gameplay-design / combat-design / executive-planning
知识库引子：`kb_search` 或 `kb_resolve_topic` 找一个真实页（如「荣耀连战」「神秘商店」），`kb_get_page` 看它实际章节。
问：
- "一份 <系统/玩法/战斗/排期> 文档，你们**必须**有哪几块？"（如系统：概述/架构/模块清单/界面流程/依赖/配表结构）
- "配表结构那节要到什么粒度——只列表名，还是要逐字段类型/取值/外键？"
- "有没有你们特有的小节（如"埋点""灰度方案""风险预案"）现在模板没有？"

### 数值类：numerical-planning
- "数值文档你们通常包含哪些：属性定义 / 成长曲线 / 公式结构 / 经济流 / 平衡校验？"
- "公式怎么表达——文字、伪代码、还是带符号定义的式子？"（**只定义结构，不填具体数值**，此红线不变）
- "有没有固定的平衡校验清单要每次附上？"

### 配表类：table-config（`assets/proposal-template.md`）+ numerical 的配表口径
知识库引子：`kb_list_tables` + `kb_get_table_schema` 看真实表字段。
- "配表变更提案要包含哪些块——改哪表 / 新增行 / 每值依据 / 校验结果 / 应用指引？现在这份缺不缺？"
- "你们看提案时，最想第一眼看到什么？"（决定提案顺序）
- "配表列口径以 `kb_get_table_schema` 为准这条不变——但提案里要不要额外标注每列的业务含义？"

### qa-review
- "你们做跨模块检查时，固定看哪几项（一致性 / 配表校验 / 模板合规 / 边界）？"
- "审阅结论要什么格式——问题清单 / 通过与否 / 风险分级？"

## 常见坑（提醒自己）

- **别把偏好当模板**：项目名、术语、默认口径、详略习惯 → 属 PROJECT_PROFILE / 系统提示词，不要塞进 output-template 的章节结构里。模板 = 骨架（有哪些章节/字段），不是内容口径。
- **别照抄知识库**：样例是引子，团队要什么由策划定。
- **别一次问太多**：一个 skill 的模板聊透就落地，避免策划疲劳导致草率点头。
- **占位符风格统一**：新章节沿用现模板的 `{...}` 占位风格。
- **定稿即出 diff**：模板一旦谈定，立刻转 skill-evolution 出 before/after diff，让策划在"真实文件改动"上做最终确认，而不是停在口头共识。
