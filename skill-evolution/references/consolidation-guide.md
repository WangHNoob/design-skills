# 固化质量指南：把经验变成好的改动

固化是本 skill 最需要判断力的一步。目标不是「把账本条目逐条翻译成补丁」，而是
**让 skill 在未来成千上万次调用中都更好用**。下面是标准与反面案例。

## 五条质量标准

### 1. 泛化，而非过拟合
账本里的经验来自具体对话，但 skill 要服务于未见过的任务。把「这次用户要的具体东西」
上升为「这类情况下应有的通用做法」。
- ✅ 「战斗设计应显式声明减伤公式采用哪种形式（除法/减法），避免口径不一。」
- ❌ 「当设计火系 BOSS 关卡3时，减伤 K 取 1000。」（一次性、过拟合、还越界填了数值）

### 2. 解释 why，而不是堆命令
今天的模型有很好的理解力。讲清楚为什么，比一串全大写 MUST/NEVER 更有效、也更好维护。
如果你发现自己在写 ALWAYS/NEVER，停一下，改成「因为……所以建议……」。
- ✅ 「先读 references/damage-formulas.md 再动手——伤害相关设计极易在减伤形式上口径不一，先对齐公式能省掉后续返工。」
- ❌ 「你必须永远先读 damage-formulas.md！！！」

### 3. 保持精简
skill 每次触发都会把 SKILL.md 正文load进上下文。臃肿的 skill 更慢、更易分散注意力。
- 能改一句就不加一段。
- **删除不再拉动效果的旧内容，也是一种进化。** 固化时顺带清理与新经验冲突或已过时的表述。
- 长而纵深、并非每次都要的知识 → 放进 references/，SKILL.md 里只留一句「何时去读」。

### 4. 尊重既有边界
进化不能破坏 skill 之间的分工和既有约束：
- numerical 之外的 skill 不填具体数值，只定义结构（这是硬约束，别在 combat/system 里写死数值）。
- 一条经验属于哪个领域就进哪个 skill，不要图方便塞进手头正在改的那个。
- knowledge-hub 为最高权威的原则、以及「无来源标注待人工补充」的原则不能被削弱。

### 5. 触发改动要连带测试
若改了 frontmatter `description` 或职责边界：
- 说明改动如何影响触发（会多触发什么 / 少触发什么）。
- 同步在 `tests/trigger-cases.json` 增补对应用例（正例和反例都要）。

## 归并策略

- 把账本里指向**同一处**的多条经验合并成一次改动，别产生多个零碎补丁。
- 相互矛盾的经验，不要都塞进去——在 diff 里把矛盾摆给用户，让他裁决。
- 低置信条目：要么这次先不动、留在账本，要么在 diff 里明确标注「低置信，请确认」。

## diff 审批环节怎么做

1. 对每个要改的文件，清楚展示 before → after（可用 Edit 的预览，或直接在回复里贴改动前后片段）。
2. 每处改动附一句「依据账本哪条 + 为什么这么改」。
3. 一次性把所有拟改动列全，让用户一览全貌再决定，而不是改一处问一次。
4. 用户可能只批准一部分——按批准的子集执行，未批准的留在账本。

## 固化后的机械步骤

不在此重复——机械步骤（写权威副本 → bump version → 同步镜像 → 验证 → 记 CHANGELOG → 归档账本）
的**唯一权威清单**在 `SKILL.md` 的「模式 B」步骤 4–9。以那里为准，避免两处描述漂移。

## 验证改动：前向测试怎么做（不泄露答案）

改完一个 skill，别只看它"读起来对不对"——要验证行为是否真的如预期改变（对"自进化"系统尤其重要，
否则会把 skill 越改越差还浑然不觉）。

- **给原始任务，不给结论**：像真实用户一样丢一个真实任务给（子）代理去做。
  - ✅ 好：「用改后的 combat-design 设计一个 3v3 实时竞技的战斗框架」——然后观察产出是否体现了这次改动。
  - ❌ 坏：「我改了减伤口径，验证它现在会用除法减伤吧」——泄露了预期答案，测试结果被污染。
- **看原始证据**：产出文件、diff、行为轨迹，而不是代理"我觉得改好了"的自评。
- **判据**：如果代理只有在你把结论告诉它之后才做对，说明改动还不够清楚，或测试已泄露答案——回到归并重写，而不是放行。
- 触发相关改动，用该 skill 的 `tests/trigger-cases.json` 做最小回归：正例仍触发、反例不误触发。
